# CareerMove PM Report — 2026-06-02

**Author:** Steve (PM, CareerMoveChecker)
**Scope:** Top 3 shippable opportunities — frontend-only or frontend + mock API, S/M effort, high user impact for job-seekers deciding "should I join this company?"

## Research Summary

### Live site investigation

The site renders as a React SPA (Vite/React Router). WebFetch only catches the `<title>` tag since the app is client-side rendered, so analysis below is derived from direct codebase reading and structural inference.

**What is built:**

| Route | Status |
|---|---|
| `/` (HomePage) | Full — hero search, AnswerCardPreview, Zones section |
| `/methodology` | Full — open scoring recipe, sources, confidence levels |
| `/app/search` | Full — search, result list, rail with bullets and bulk CTA |
| `/app/company/:id` | Full — Zone A (identity), Zone B (verdict card), Zone C (5-tab deep-dive: Risk flags, Identity, People, Finance, Filings) |
| `/app/saved` (Dashboard) | Full — saved list, alerts feed, filter/sort/select |
| `/app/compare` | Full — up to 3 companies side-by-side |
| `/app/bulk` | Full — CSV upload, 3-step flow, results table |
| `/pricing` | **Missing — route not in router.tsx** |

**UX gaps identified from code:**

1. **No pricing page exists.** `router.tsx` has no `/pricing` route. The MarketingNav (`MarketingNav.tsx:4-20`) only contains a logo and a single "Check a company" CTA — no pricing, no sign-in, no plans. The MarketingFooter (`MarketingFooter.tsx:20-28`) links to Search, Compare, Saved, Example, Methodology — but no Pricing. Users who want to understand the paid tiers (Pro/Agency) have no dedicated page; the only entry point to those tiers is the `WaitlistCTA` / `WaitlistModal` component, which surfaces inline on unnamed pages.

2. **No persona switcher in the app UI.** The persona system (candidate / freelancer / agency) exists in `persona.tsx` and `persona-copy.ts` but there is **no UI to switch persona** visible in `AppNav.tsx` or any page. The persona defaults to `'candidate'` from localStorage. Users who arrive as freelancers or agencies never see copy tuned for them. The `SEARCH_RAIL_BULLETS` and `REPORT_QUESTION` are hardcoded to single universal strings — the persona branching logic appears to have been simplified away but the infrastructure remains dormant.

3. **Director cross-check signal is described but not shown.** The Methodology page (`MethodologyPage.tsx:17`) documents `/officers/{id}/appointments` as a "Director cross-appointments — 'killed companies before' signal". The risk flag system in `types/index.ts:78-85` supports this via `RiskFlag` with evidence and recommendedAction. However, in `CompanyReportPage.tsx:188-197`, `topReasons` are shown but **no dedicated "Director history" section** is rendered in any tab — officers tab (`tab === 'people'`) shows name, role, and dates only (`CompanyReportPage.tsx:287-308`), with zero cross-appointment or disqualification context beyond the top-level `disq-banner`. Competitors (Red Flag Alert, Creditsafe) surface director track record as a primary risk signal.

4. **Bulk check has no CSV export of results.** The `BulkCheckPage.tsx` toolbar (`line 118-123`) has filter chips but no "Export results" button. Users who upload 50 companies get verdicts on-screen but cannot download them. The `SavedPage.tsx` bulk toolbar (`line 193-199`) shows "Export PDF" button but it has no `onClick` handler — it is a dead button. `api/client.ts` has no export endpoint defined.

5. **No feedback loop on the report page.** `api/client.ts:61-67` defines `submitFeedback` with rating, useCase, and comment fields, but there is **no feedback widget rendered anywhere in `CompanyReportPage.tsx`**. The endpoint is wired but the UI that would call it was never built.

6. **Marketing nav is extremely sparse.** `MarketingNav.tsx` is 22 lines with only a logo and one CTA. No Methodology link, no Pricing link. Users who want to understand how scores work before trying the tool have no path from the nav.

### Competitor analysis

| Competitor | Key differentiator over basic CH |
|---|---|
| **Companies House** (official) | Raw filing data, no interpretation, prominently disclaims accuracy. UX is gov-standard minimal. |
| **Red Flag Alert** | Real-time credit risk + AML/KYC + portfolio monitoring + CRM integrations. B2B sales-oriented. Director screening is a named feature. |
| **Creditsafe** | Credit scores with recommended credit limits, CCJ history, payment behaviour, global coverage, "predictive risk modelling". "#1 on Trustpilot" trust signal. |
| **Endole** | Unreachable during research session. |
| **Crunch** | 404 on targeted URL. |

**What competitors do that CareerMove does not (yet):**
- Creditsafe and Red Flag Alert both show a **numeric credit/risk score with a visual indicator** prominently on the result card (not buried in a tab).
- Red Flag Alert surfaces **director appointment history across multiple companies** as a named feature — "has this director killed companies before?" is their tagline language.
- Creditsafe shows **CCJ (County Court Judgement) records** — a major signal for freelancers worried about being paid.
- Both B2B tools provide **"download as PDF/report"** for every result — critical for sharing with colleagues before signing an offer.

## Top 3 Opportunities

---

### Opportunity 1: Director history cross-check panel in the People tab

**Problem**

The report's People tab (`CompanyReportPage.tsx:281-309`) shows officer name, role, and dates. It does not surface the most compelling signal CareerMove claims to compute: whether a director has previously been associated with dissolved, struck-off, or insolvent companies. The Methodology page (`MethodologyPage.tsx:16`) documents `/officers/{id}/appointments` as the source for the "killed companies before" signal, and the `RiskFlag` type (`types/index.ts:78-85`) fully supports evidenced flags with `recommendedAction`. But in the People tab today, the officer card is a flat list with no cross-appointment data whatsoever.

**Evidence**

- `frontend/src/pages/CompanyReportPage.tsx:287-308` — officer card renders name, role, occupation, and appointed/resigned date only.
- `frontend/src/pages/MethodologyPage.tsx:16` — explicitly claims cross-appointment signal is computed.
- `frontend/src/types/index.ts:24-36` — `Officer` type has no `previousCompanies` or `appointments` field; this data is not yet in the mock API contract.
- `frontend/src/api/client.ts:29-95` — no endpoint for fetching officer appointments separately.
- Competitor: Red Flag Alert and Creditsafe both name "director track record" as a primary feature and surface it directly on the result card.
- Job-seeker use case: The Methodology `cant-grid` (`MethodologyPage.tsx:194-204`) explicitly says CareerMove *can* answer "Has the director killed companies before?" — so the user expects it on the report, not just in the methodology.

**Proposed solution**

1. Extend the `Officer` type (`types/index.ts:24-36`) to add an optional `previousCompanies?: { name: string; number: string; status: string; ceasedOn?: string }[]` field.
2. Add a mock API response that populates `previousCompanies` for at least one officer in the existing mock data.
3. In the People tab of `CompanyReportPage.tsx`, expand the officer card to show a collapsible "Director history" row: for each officer with previous companies, render a compact table of their other/past appointments with status badges (dissolved/active). Use the existing `badge-ok` / `badge-bad` badge system already in the codebase.
4. Add a summary line at the top of the People tab: e.g. "1 of 3 directors has been associated with a dissolved company — see below."
5. Source tag the section as `zone-tag deduced` (matching the existing UI pattern from `CompanyReportPage.tsx:249`).

**User impact**

This is the single question CareerMove most loudly promises to answer ("Has the director killed companies before?") and currently fails to deliver on the report page. Showing it turns the People tab from a phone-book list into a real risk signal for job-seekers. It directly addresses the job-seeker's core fear: "Is this founder/director trustworthy?"

**Effort:** M (extend type + mock data + 40-60 lines of JSX in the People tab panel)

**Confidence:** 9/10

### Opportunity 2: Inline risk score gauge on the verdict card (Zone B)

**Problem**

The verdict card (`CompanyReportPage.tsx:170-227`) shows a headline verdict ("Probably yes." / "Caution." / "No — serious risk."), a prose explanation, and up to 4 tick items. It does not render the numeric `assessment.score` (0-100) or `assessment.confidence` (0-1) that already exist in the `RiskAssessment` type (`types/index.ts:93-104`). The score is used only as a sort key in `SavedPage.tsx:239` (`${r.assessment.score}/100` in a tiny `span.small`).

Competitors Creditsafe and Red Flag Alert lead with a visual risk indicator (gauge, colour band, or numbered score). At-a-glance trust calibration is their primary UI pattern. CareerMove's verdict card is text-heavy and requires reading prose to calibrate confidence.

**Evidence**

- `frontend/src/types/index.ts:93-104` — `score: number` (0-100), `confidence: number` (0-1), and `riskLevel: RiskLevel` all exist.
- `frontend/src/pages/CompanyReportPage.tsx:170-227` — Zone B answer card renders no score visual, only text.
- `frontend/src/pages/SavedPage.tsx:239` — the only place score is shown is a tiny inline string: `${r.assessment.score}/100`.
- `frontend/src/pages/BulkCheckPage.tsx:147-150` — the bulk result table shows a confidence bar widget that works well: `<div class="bar"><i style={{ width: ... }}></div>`. This component could be reused.
- Competitor: Creditsafe leads every result with a colour-banded credit score circle. Red Flag Alert uses a traffic-light indicator above the fold. Both are shown before any prose.
- The Methodology page (`MethodologyPage.tsx:25-42`) documents that the score starts at 75 and deducts/adds — users who read methodology expect to see the number on the report.

**Proposed solution**

1. In `CompanyReportPage.tsx` Zone B answer card, add a score gauge immediately after the disqualification banner and before `answer-q`. Render a horizontal colour bar (green/amber/red based on score) with the numeric score (e.g. "72 / 100") and confidence level (HIGH / MODERATE / LOW derived from `assessment.confidence`) as a chip.
2. Reuse the confidence bar markup already in `BulkCheckPage.tsx:147-150` — extract it to a shared `<ConfidenceBar />` component in `frontend/src/components/`.
3. Colour the gauge using existing CSS variables: `--ok` (green, score ≥ 70), `--warn` (amber, 40-69), `--bad` (red, < 40).
4. Add a tooltip/link to the Methodology page with "How is this calculated?" — wires back to the open-scoring principle already documented.
5. No new API endpoints needed — data is already in `CompanyReport.assessment`.

**User impact**

A numeric score with colour band gives a job-seeker instant calibration before reading prose. It reduces cognitive load ("I can see 72/100 green before I read the paragraphs"), makes reports shareable ("the company scored 38/100"), and matches the mental model set by competitors and by CareerMove's own Methodology page. It also makes the SavedPage table and Compare page more legible by establishing what "72" means.

**Effort:** S (extract shared component, ~30 lines JSX, CSS colour logic already exists as variables)

**Confidence:** 9/10

### Opportunity 3: Shareable / printable report summary card with one-click copy link

**Problem**

A job-seeker's primary action after checking a company is sharing the result — with a partner, a recruiter, a friend in their network, or their own notes. CareerMove currently has a Print button (`CompanyReportPage.tsx:150`) and a "Copy invoice block" button (`CompanyReportPage.tsx:211-223`), but no way to share the report itself as a link or as a portable summary. The "Copy invoice block" is clearly a freelancer/B2B feature, not a job-seeker feature. There is no "Share" or "Copy link" action.

The report URL is already canonical (`https://careermove.uk/c/{companyNumber}` per `CompanyReportPage.tsx:38`) and schema.org JSON-LD is injected (`CompanyReportPage.tsx:41-64`), so deep-linking is technically supported. But there is no affordance in the UI to copy that URL, and no summary card designed for paste-into-chat sharing.

**Evidence**

- `frontend/src/pages/CompanyReportPage.tsx:146-159` — `id-actions` row shows: Refresh, Print, Compare. No Share/Copy Link.
- `frontend/src/pages/CompanyReportPage.tsx:38` — canonical URL is already `https://careermove.uk/c/{companyNumber}`.
- `frontend/src/pages/CompanyReportPage.tsx:41-64` — schema.org metadata injected — good for link previews.
- `frontend/src/pages/CompanyReportPage.tsx:211-223` — "Copy invoice block" copies name + address, not verdict or URL.
- The Save CTA (`CompanyReportPage.tsx:199-209`) says "Save this company. Get notified if anything changes." — but saving is account-gated (requires the API to persist) whereas sharing just requires a URL copy.
- Competitor pattern: Endole, Creditsafe, and Companies House all return stable, copyable URLs per company. Red Flag Alert generates PDF exports.
- Job-seeker context: "I got an offer from Company X, here's what CareerMove says about them" is an obvious WhatsApp/Slack share scenario.

**Proposed solution**

1. Replace the "Copy invoice block" button in Zone B (which is freelancer-specific and buried) with two buttons:
   - **"Copy link"** — copies `https://careermove.uk/c/{companyNumber}` to clipboard, shows the existing toast "Link copied" using the established toast pattern (`CompanyReportPage.tsx:115-129`). Implementation: `navigator.clipboard.writeText(canonicalUrl)` — identical to the existing invoice-block copy pattern.
   - **"Share summary"** — copies a rich-text block: company name, number, risk level, score, and the one-sentence verdict. Format: `{CompanyName} (#number) — CareerMove verdict: {riskLevel} ({score}/100). {verdict.slice(0,120)}… https://careermove.uk/c/{number}`
2. Move "Copy invoice block" to the Finance tab where it is contextually appropriate for freelancers (next to the Charges section).
3. Add an "Open in Companies House" link to the `id-actions` bar (linking to `https://find-and-update.company-information.service.gov.uk/company/{companyNumber}`) — this reinforces the "Direct from CH" trust message and is listed as a redirect target in Methodology already.

**User impact**

Enables the most natural sharing behaviour for job-seekers (and for recruiters using the tool). A shared link with og:title/description (already supported via schema.org + `useSeo`) creates a rich preview in Slack/WhatsApp showing the company verdict — free viral distribution. The summary copy-paste allows sharing in contexts where a link preview doesn't render. This is zero-backend work: clipboard API, existing toast system, and the canonical URL that already exists.

**Effort:** S (< 20 lines of change in CompanyReportPage.tsx, no new components needed)

**Confidence:** 10/10

## Priority Order

| Rank | Opportunity | Effort | Confidence | User Impact |
|---|---|---|---|---|
| 1 | Share / Copy link (Opp 3) | S | 10 | Viral growth + immediate user utility |
| 2 | Risk score gauge on verdict card (Opp 2) | S | 9 | Trust calibration before reading prose |
| 3 | Director history cross-check panel (Opp 1) | M | 9 | Fulfils core promise; differentiates from CH raw data |

Opportunities 1 and 2 are both ranked high but Opp 3 is listed first because it is pure polish on existing data with no type changes — it ships fastest and is the most likely to drive organic sharing (the primary growth lever for a free B2C tool). Opp 2 follows because it closes a gap between the promise (Methodology page shows the score formula) and the delivery (score not visible on the report). Opp 1 is ranked third because it requires extending the type contract and mock data, but it is the highest *differentiation* opportunity — no competitor in the free tier shows director appointment history this clearly.

## What was not selected (and why)

- **Pricing page** — important structurally (no route exists, nav has no link) but it is primarily a conversion/monetisation concern, not a job-seeker usability concern. Should ship, but it is more a marketing/sales page than a product feature.
- **PDF export from Saved / Bulk** — the "Export PDF" button in `SavedPage.tsx:197` is dead (no handler). Wiring it would require either a backend endpoint or a client-side print stylesheet. Medium effort with unclear immediate job-seeker value vs. the above three.
- **Feedback widget** — `api/client.ts:61-67` has the endpoint but no UI. Good for data collection but does not directly help the user make a decision.
- **Persona switcher UI** — the persona system is dormant in the UI. Re-activating it would increase copy relevance but the universal copy is already good. Lower priority than filling product gaps.

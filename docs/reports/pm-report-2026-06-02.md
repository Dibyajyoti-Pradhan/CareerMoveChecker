### #1 — Add a visible `ScoreGauge` to the report Zone B verdict card

**Priority:** 1

The `ScoreGauge` component already exists at `components/ScoreGauge.tsx` and is used in both `CompanyReportPage.tsx` (not in Zone B) and `BulkCheckPage.tsx`. It renders a `role="meter"` bar with the score fraction and a link to the Methodology page.

**User impact**

Every visitor to a company report gets an instant visual trust calibration before reading prose. Reduces time-to-verdict for repeat users. Aligns with the mental model set by the Methodology page. Makes reports more shareable ("scored 38/100").

### #2 — Fix missing/silent API error states on Search and Report pages

**Priority:** 2

**Problem**

Two critical flows swallow API errors and display misleading UI:

1. **SearchPage** (`SearchPage.tsx:51–56`): The `.catch()` block sets `hits = []` and `loading = false`, causing the `noResults` state to display "No companies found for '…'" even when the real problem is a network failure or a 500 from the server. There is no `error` state variable.

2. **CompanyReportPage** (`CompanyReportPage.tsx:77–84`): `api.getReport(id)` is called with `.catch(() => null)` directly in the client (`api/client.ts:36`). If the API fails, `report` is set to `null`, and the "Company not found" state card renders — identical to a genuinely missing company. A user who gets a 503 sees "We couldn't find {id} at Companies House" and has no reason to try again.

3. **ComparePage** (`ComparePage.tsx:21–24`): `api.compare` catch sets `loading = false` with no error state, silently collapsing to the "Add a company number" empty state.

**Proposed solution**

For each affected page:
- Add an `error: string | null` state variable initialised to `null`.
- In the `.catch()` block, set `error` to a user-friendly message (e.g. `'Could not reach Companies House — please try again.'`).
- Render a distinct error UI (reuse the existing `.state-card.danger` CSS class from `pages.css:361`) with a Retry button that re-triggers the fetch.
- Keep the "no results" state for the case where the API returns `[]` or a 404.

This is a 3-file change of ~10–15 lines each, with no new components needed.

**User impact**

Eliminates false "company not found" messages during outages. Gives users a retry path instead of a dead end. Builds trust in data quality — users who know an error occurred will re-try; users who see "not found" will assume the company doesn't exist and leave.

**Effort:** S  
**Confidence:** High

### #3 — Fix tab accessibility (ARIA roles, keyboard navigation) on the Company Report

**Priority:** 3

**Problem**

The Zone C tab strip in `CompanyReportPage.tsx:267–313` renders five tab buttons and conditional panels. The tabs have no ARIA attributes:

- `<button className="tab">` — missing `role="tab"`, `aria-selected`, `aria-controls`
- The panel `<div>` for each tab — missing `role="tabpanel"`, `aria-labelledby`
- The container `.tabs` — missing `role="tablist"`
- No keyboard `ArrowLeft` / `ArrowRight` navigation between tabs

This violates WCAG 2.1 SC 4.1.2 (Name, Role, Value) and the ARIA authoring practices for tab widgets. Companies House, Endole, and gov.uk design system all use correct ARIA tab patterns. Screen-reader users cannot navigate the report.

**Proposed solution**

1. Add `role="tablist"` to `.tabs` container.
2. For each `<button className="tab">`, add:
   - `role="tab"`
   - `aria-selected={tab === t}`
   - `id={`tab-${t}`}`
   - `aria-controls={`panel-${t}`}`
3. For each conditional panel `<div>`, add:
   - `role="tabpanel"`
   - `id={`panel-${t}`}`
   - `aria-labelledby={`tab-${t}`}`
   - `tabIndex={0}` (so panel receives focus on tab switch)
4. Add a `keydown` handler on the `tablist` for `ArrowLeft` / `ArrowRight` to cycle active tab.

This is a self-contained change within `CompanyReportPage.tsx` (~25 lines).

**User impact**

Makes the report usable for screen-reader users and keyboard-only users. Required for WCAG 2.1 AA compliance. Also improves SEO tab switching discoverability. The fix demonstrates CareerMove's commitment to accessibility — important trust signal for enterprise/agency buyers who must meet procurement accessibility requirements.

**Effort:** S  
**Confidence:** High

### #6 — Add `aria-valuetext` to ScoreGauge and improve score colour contrast

**Priority:** 6

**Problem**

`ScoreGauge.tsx` renders a `role="meter"` element with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`, and `aria-label="Risk score"`. However, `aria-valuetext` is absent. Screen-readers will announce "Risk score, 72" with no colour or risk-level context. WCAG 2.1 SC 1.3.1 requires that the semantic role communicate the meaning — a raw number without context ("72 out of 100, LOW risk") is insufficient.

Additionally, the confidence text `Confidence: {Math.round(confidence * 100)}%` is rendered in `color: var(--muted)` (CSS variable `#5b6473` at `index.css:14`). On a white background, this gives a contrast ratio of approximately 4.3:1 — below the 4.5:1 threshold required by WCAG SC 1.4.3 for normal-weight text at 13px.

**Proposed solution**

In `ScoreGauge.tsx`:

1. Derive a `riskLabel` from the score: `score >= 70 ? 'LOW risk' : score >= 40 ? 'MODERATE risk' : 'HIGH risk'`.
2. Add `aria-valuetext={`${score} out of 100 — ${riskLabel}, confidence ${Math.round(confidence * 100)}%`}` to the meter div.
3. For the confidence span, change `color: var(--muted)` to `color: var(--ink-2)` (`#1f2937`, contrast ratio ~11.9:1) or add a font-weight of 500 to pass contrast at the smaller size.

This is a 5-line change in one component that also affects `BulkCheckPage.tsx` (which reuses `ScoreGauge`).

**User impact**

Screen-reader users get meaningful score announcements. Fixes a WCAG 1.3.1 + 1.4.3 violation. Benefits visually impaired job-seekers and enterprise clients who audit accessibility before procurement.

**Effort:** S  
**Confidence:** High

### #7 — Fix the mobile layout: stack the search rail below results, show saved-table columns on mobile

**Priority:** 7

**Problem**

Two critical mobile layout regressions exist in `pages.css`:

**A. Search page rail disappears on mobile.**
The `.search-grid` at `pages.css:319` is `grid-template-columns: 1fr 320px`. At ≤1000px (`pages.css:662`), the responsive rule collapses `.search-grid` to `grid-template-columns: 1fr` — but the `.rail` (right column) has no explicit `display` or `order` override. The rail becomes the second grid item and stacks below the results, which is correct in theory, but in practice the `@media` rule at `pages.css:662` lists `.search-grid` in the comma-separated collapse list, meaning the rail simply stacks — however, the `.rail-card.quota-card` Bulk Check CTA becomes invisible below a long result list, defeating its purpose as a conversion driver.

**B. Saved-table hides all meaningful columns on mobile.**
`pages.css:670`: `.saved-table .head, .srow { grid-template-columns: 38px 1fr }` and then `:not(:nth-child(-n+2))` hides all columns after the checkbox and company name. This means Status, Trust Verdict, and Risk Level are hidden — the table shows only a name with a checkbox. The user cannot assess risk at a glance and must click into each company.

**Proposed solution**

A. For the search rail: add `order: -1` to `.rail` inside the `@media (max-width: 1000px)` block so the "What you'll learn" and "Bulk check" cards appear **above** the result list on mobile — where they act as onboarding guidance before the user sees results.

B. For the saved-table on mobile: change the collapsed template to `38px 1fr auto` (add one auto column for the risk pill), and change the `:not(:nth-child(-n+2))` rule to `:not(:nth-child(-n+3))` to preserve the Trust Verdict column. This gives mobile users the company name plus the coloured OK/WATCH/AVOID pill — enough for triage.

**User impact**

Mobile users (likely 40–60% of traffic for a tool used when reviewing job offers on the go) can use the Saved dashboard for triage. The Bulk CTA remains visible on mobile Search, preserving the conversion path to paid tiers.

**Effort:** S (CSS-only, ~8 lines in `pages.css`)  
**Confidence:** High

## Ranking rationale

Items 1–4 are all **S effort, High confidence** and address gaps that every visitor to the core report page will encounter. They are ordered by breadth of user impact: the score gauge affects every report view (1), error states affect every API failure (2), tab accessibility affects keyboard/screen-reader users and is a compliance requirement (3), and the copy-link button is a zero-risk addition that enables viral sharing (4).

Item 5 is **M effort** but is listed at #5 because it delivers on the single most prominent product promise ("Has the director killed companies before?") — the gap between the methodology documentation and the actual UI is the most significant credibility risk.

Items 6–8 are quick S-effort fixes that close accessibility and mobile gaps introduced by CSS rules in `index.css` and `pages.css`.

Items 9–10 are **M effort** but lower priority: the print stylesheet improves an existing feature rather than unlocking new user behaviour, and the pricing page nav link is important for conversion but does not affect the core report-reading flow.

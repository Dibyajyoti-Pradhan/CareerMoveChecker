## Executive Summary

CareerMove has a well-structured frontend — clean component separation, ARIA-aware tab navigation, print support, and meaningful data types. The gap between what the product *could* show and what it *does* show is the dominant theme. The codebase defines rich typed data (explanationSummary, SIC code descriptions, disqualification detail, address region) that never reaches the screen; there are keyboard and ARIA gaps that harm accessibility; and the compare and search pages have minor but impactful UX rough edges that erode trust at the exact moment a user is deciding whether to rely on the verdict.

Competitors (Red Flag Alert, Creditsafe, Endole) win on: (1) persistent "what changed" narratives not just alerts, (2) inline context for raw codes like SIC, and (3) shareable, self-contained report views. All three are achievable here without backend changes.

Six items are S-effort (under 2h); four are M-effort (2–6h). None require a new API endpoint or backend change.

## Item 1 — `explanationSummary` is typed and delivered but never rendered

**Problem statement:** The `RiskAssessment` type declares `explanationSummary: string` — a longer narrative explaining the engine's reasoning. It arrives in every `getReport` and `compare` response. Nothing in the UI ever renders it. The assessment section shows only the short `verdict` string and up to four `topReasons` bullets. The fuller narrative — the thing that most distinguishes CareerMove from a raw Companies House dump — is silently dropped.

**Evidence:**
- `frontend/src/types/index.ts` line 108: `explanationSummary: string;` defined on `RiskAssessment`
- `frontend/src/pages/CompanyReportPage.tsx` lines 299–302: Zone B renders `a.verdict` then `a.topReasons.slice(0,4)`. The `a.explanationSummary` field is never accessed in this file or any other page file.
- A full grep across all `src/` files for `explanationSummary` yields zero usage outside the type definition.

**Proposed solution:** In `CompanyReportPage.tsx`, inside Zone B, immediately after the `answer-verdict` paragraph (line 301), add:

```tsx
{a.explanationSummary && (
  <p style={{ fontSize: 14, color: 'var(--ink-2)', lineHeight: 1.6, marginTop: 8, marginBottom: 0 }}>
    {a.explanationSummary}
  </p>
)}
```

The field is already gated on the `CompanyReport` type so no mock-API change is needed. The block already exists in the rendered code — this is a 3-line addition.

**User impact:** Every user who opens a report receives a richer plain-English explanation. Directly reinforces the core brand promise ("no jargon"). Reduces bounce from the answer card because the verdict no longer reads as thin. Affects 100% of report views.

**Effort:** S (< 2h)
**Confidence:** 5/5

## Item 2 — Search result rows are click-only; keyboard users cannot open them

**Problem statement:** `ResultRow` in `SearchPage.tsx` renders search results as a `<div onClick={onOpen}>`. There is no `role="button"`, `tabIndex`, or `onKeyDown` handler. A keyboard-only user (Tab to navigate, Enter to activate) can reach the "Open report" link inside the row but cannot activate the row itself. This also means the whole-row click target — which is the primary intended interaction — is inaccessible.

**Evidence:**
- `frontend/src/pages/SearchPage.tsx` line 229: `<div className="result" onClick={onOpen} style={{ cursor: 'pointer' }}>` — a plain `div` with a click handler.
- No `tabIndex`, no `role`, no `onKeyDown` anywhere in `ResultRow`.
- The `<Link>` "Open report" button at line 243 is reachable by Tab but it is a secondary affordance styled as a small secondary button, not the primary row target.

**Proposed solution:** Convert the outer `div` to be keyboard-accessible:

```tsx
<div
  className="result"
  onClick={onOpen}
  onKeyDown={(e) => { if (e.key === 'Enter' || e.key === ' ') { e.preventDefault(); onOpen(); } }}
  role="button"
  tabIndex={0}
  style={{ cursor: 'pointer' }}
  aria-label={`Open report for ${hit.companyName}`}
>
```

This is a 4-attribute addition to a single element. The `<Link>` "Open report" inside can keep `e.stopPropagation()` to prevent double-fire.

**User impact:** Keyboard-only users (power users, users with motor disabilities, QA testers) can now navigate the search result list natively. Also resolves a WCAG 2.1 SC 2.1.1 violation. Affects every search on the app.

**Effort:** S (< 2h)
**Confidence:** 5/5

## Item 3 — SIC codes are shown as raw numbers with no human-readable label

**Problem statement:** The company report's "Identity & address" section (Zone A id-meta) and the compare grid both render SIC codes as raw numeric codes (e.g. `62020`, `56101`). SIC codes are meaningless to most users without a lookup. Every competitor (Creditsafe, Endole) resolves the code to a human label inline. A user checking a company before a contract or job offer needs to know the industry at a glance.

**Evidence:**
- `frontend/src/pages/CompanyReportPage.tsx` line 259: `<span className="val">{p.sicCodes?.join(', ') || '—'}</span>` — raw codes only.
- `frontend/src/pages/ComparePage.tsx` line 212: `val: r.profile.sicCodes?.join(', ') || '—'` — same.
- Endole and Creditsafe both show "Software Consulting (62020)" or equivalent; Red Flag Alert shows sector labels instead of codes.

**Proposed solution:** Add a static SIC-to-label lookup object in `frontend/src/lib/format.ts` covering the ~100 most common UK SIC codes (the CH public list is finite and stable). Expose a `formatSicCode(code: string): string` helper that returns `"62020 · Computer programming"` or falls back to the raw code if unknown. Apply it in `CompanyReportPage.tsx` and `ComparePage.tsx`.

```ts
// lib/format.ts addition
const SIC_LABELS: Record<string, string> = {
  '62020': 'Information technology consultancy',
  '56101': 'Licensed restaurants',
  '64110': 'Central banking',
  // ... ~100 entries covering 90%+ of real companies
};
export function formatSicCode(code: string): string {
  return SIC_LABELS[code] ? `${code} · ${SIC_LABELS[code]}` : code;
}
```

No API change needed. The SIC master list is public domain data (~6KB of JSON).

**User impact:** Every user reading a report immediately understands the company's industry without leaving the page. Removes a key friction point for the "can I trust this company before signing an offer?" use case. Medium-to-high impact for all report views.

**Effort:** M (2–6h to compile the SIC lookup table and wire it in both pages)
**Confidence:** 5/5

## Item 4 — Compare page verdict text is hard-truncated at 80 characters mid-sentence

**Problem statement:** In `ComparePage.tsx`, the `Verdict` row's `sub` field is built as `r.assessment.verdict.slice(0, 80) + '…'`. This always appends an ellipsis even for verdicts shorter than 80 characters, and truncates mid-word for verdicts that happen to be slightly longer. The cut-off is arbitrary and often produces unreadable sub-text in the comparison grid (e.g. "Strong public signals. 11 years of continuous trading, no insolvency on file, accou…").

**Evidence:**
- `frontend/src/pages/ComparePage.tsx` line 201: `sub: r.assessment.verdict.slice(0, 80) + '…'` — unconditional truncation.
- A 75-character verdict still gets `…` appended. A 90-character verdict is cut mid-word.

**Proposed solution:** Replace the hard truncation with a conditional smart truncation using the existing `formatDate`/`relativeTime` pattern in `format.ts`. A one-line fix in `ComparePage.tsx`:

```ts
sub: r.assessment.verdict.length > 80
  ? r.assessment.verdict.slice(0, 80).replace(/\s\S*$/, '') + '…'
  : r.assessment.verdict,
```

This (a) only appends ellipsis when the string genuinely exceeds 80 chars, and (b) truncates at a word boundary.

**User impact:** Users comparing companies no longer see mid-word truncated verdict text. Particularly important for the "Watch" and "Caution" verdicts where the sub-text carries the meaningful nuance. Small change, high polish payoff.

**Effort:** S (< 2h — single line change plus one regression test)
**Confidence:** 5/5

## Item 5 — Feedback widget on report page is binary thumbs-up/down with no text path

**Problem statement:** The feedback row at the bottom of `CompanyReportPage.tsx` offers only 👍 / 👎. The API `submitFeedback` endpoint already accepts an optional `comment?: string` field (defined in `api/client.ts` line 65 and typed in the params). No UI exposes this. Competitors (and SaaS review patterns broadly) show significantly higher feedback completion when a follow-up text field appears on 👎. Without it, negative feedback is a black hole — the team gets a rating but zero context.

**Evidence:**
- `frontend/src/api/client.ts` lines 62–67: `comment?: string` field typed but never sent.
- `frontend/src/pages/CompanyReportPage.tsx` line 664 and 679: both `submitFeedback` calls pass `{ companyNumber, rating, useCase }` only — `comment` is always absent.
- The feedback row UI has no text input, no expand state.

**Proposed solution:** On 👎 click: instead of immediately submitting, transition the feedback row to show a one-line `<textarea>` pre-labelled "What could be better? (optional)" with a "Submit" button. On 👍, submit immediately as now (high-rating feedback rarely needs context). This reuses the existing `feedbackState` state machine by adding a `'commenting'` state:

- `'idle'` → user clicks 👎 → state = `'commenting'`
- `'commenting'` → shows textarea + Submit button → on submit → state = `'submitting'` → `'done'`
- `'idle'` → user clicks 👍 → state = `'submitting'` → `'done'`

No new API endpoint. The `comment` field already exists.

**User impact:** Product team gets actionable negative feedback instead of a bare count. Estimated 2–3× improvement in qualitative signal from negative ratings based on standard SaaS UX patterns. Low user cost (textarea is optional, collapses cleanly).

**Effort:** M (2–6h)
**Confidence:** 4/5

## Item 6 — Bulk check page has no keyboard-accessible drag-and-drop zone

**Problem statement:** `BulkCheckPage.tsx` renders a drag-drop upload zone as a styled `<div>` with `onDragOver`, `onDrop`, and `onDragLeave` handlers. There is no `tabIndex`, `role`, `aria-label`, or `onKeyDown` on this element. Keyboard users see the drop zone visually but cannot interact with it — only the hidden `<input type="file">` inside the "Upload CSV" button is accessible, and that button is in the `page-head` area far above the drop zone. The disconnect between the visual drop-zone affordance and its inaccessibility is confusing.

**Evidence:**
- `frontend/src/pages/BulkCheckPage.tsx` lines 253–271: The drop zone `<div>` has drag events but no keyboard or ARIA attributes.
- `frontend/src/pages/BulkCheckPage.tsx` lines 104–107: The accessible upload path (`<label>` wrapping `<input type="file">`) is only in the page header, not associated with the drop zone.

**Proposed solution:**
1. Add `tabIndex={0}`, `role="button"`, `aria-label="Upload CSV file — click or press Enter to browse"` to the drop zone `<div>`.
2. Add `onKeyDown={(e) => { if (e.key === 'Enter' || e.key === ' ') { e.preventDefault(); hiddenInputRef.current?.click(); } }}`.
3. Add a `useRef<HTMLInputElement>(null)` pointed at a hidden `<input type="file">` inside the drop zone (or reuse the existing one with a forwarded ref).

**User impact:** Keyboard and assistive-technology users can now access the primary bulk-check workflow. Resolves WCAG 2.1 SC 2.1.1. Also improves perceived quality for power users who prefer keyboard-first interaction.

**Effort:** S (< 2h)
**Confidence:** 5/5

## Item 7 — `⌘K` jump-to-search hint is shown in the search UI but the shortcut is only wired in AppNav — not on marketing pages

**Problem statement:** `SearchPage.tsx` line 102 displays `<kbd>⌘K</kbd> jump to search` as a keyboard shortcut hint. The handler for `⌘K` is in `AppNav.tsx` (line 12–18) which navigates to `/app/search`. This works inside the app shell. However, on the marketing pages (`/`, `/pricing`, `/methodology`) there is no `AppNav` — there is `MarketingNav` — and the `⌘K` handler is never registered. A user on the home page who presses `⌘K` (a common pattern) gets nothing. The hint is not shown on marketing pages, but the broken expectation affects users who know the shortcut from inside the app.

**Evidence:**
- `frontend/src/layout/AppNav.tsx` lines 11–18: `⌘K` handler registered only inside `AppNav`.
- `frontend/src/layout/MarketingNav.tsx`: 22 lines, no keyboard handler.
- `frontend/src/pages/HomePage.tsx`: The hero search form has no global keyboard listener.

**Proposed solution:** Extract the `⌘K` handler into a small custom hook `useGlobalSearch()` in `lib/` that can be called from both `AppNav` and `MarketingNav` (or from `main.tsx` as a singleton). A 10-line hook:

```ts
// lib/useGlobalSearch.ts
export function useGlobalSearch() {
  const navigate = useNavigate();
  useEffect(() => {
    const h = (e: KeyboardEvent) => {
      if ((e.metaKey || e.ctrlKey) && e.key === 'k') {
        e.preventDefault();
        navigate('/app/search');
      }
    };
    window.addEventListener('keydown', h);
    return () => window.removeEventListener('keydown', h);
  }, [navigate]);
}
```

Call it in `MarketingLayout.tsx` alongside the existing `AppNav` call.

**User impact:** Power users (the target audience — contractors, freelancers, agency staff) rely on `⌘K` patterns. Bringing the shortcut to the marketing entry point reduces navigation friction for returning users who land on the home page. Consistent with the product's "built for professionals" positioning.

**Effort:** S (< 2h)
**Confidence:** 4/5

## Item 8 — Saved companies page makes N parallel `getReport` calls on load with no rate-limit protection or loading skeleton per row

**Problem statement:** `SavedPage.tsx` lines 49–56 load every saved company's full report in parallel via `Promise.all(r.map(async (s) => api.getReport(s.companyNumber)))`. With the free tier cap of 10 saved companies this fires 10 simultaneous `/api/companies/{n}/report` calls on page load. There is no debounce, no sequential batching, and no per-row loading state — the table shows `'loading'` as the verdict text until all N calls complete. In the real backend, concurrent CH API calls at this scale will hit rate limits. In the mock, the pattern also means the page's useful state arrives as a single all-or-nothing flush rather than progressively.

**Evidence:**
- `frontend/src/pages/SavedPage.tsx` lines 49–56: `Promise.all(r.map(...api.getReport...))` on every page mount.
- Line 279: Each row shows `r ? '${r.assessment.score}/100' : 'loading'` — no per-row skeleton, just the text "loading".
- The API client in `client.ts` has no concurrency control.

**Proposed solution:** Replace `Promise.all` with a sequential-with-parallelism-cap pattern using a simple async queue (max 3 concurrent):

```ts
async function fetchReportsInBatches(
  numbers: string[],
  onResult: (n: string, r: CompanyReport | null) => void,
  concurrency = 3
) {
  let i = 0;
  const workers = Array.from({ length: concurrency }, async () => {
    while (i < numbers.length) {
      const n = numbers[i++];
      const r = await api.getReport(n).catch(() => null);
      onResult(n, r);
    }
  });
  await Promise.all(workers);
}
```

Call `onResult` to incrementally set `reports` state so each row renders its score as soon as its fetch completes. Add a per-row skeleton (`<div className="skel" style={{width:60,height:18}} />`) while its report is pending.

**User impact:** (a) Backend safety — reduces peak CH API load from N concurrent to max 3. (b) UX — users see verdicts appear row-by-row rather than all at once or not at all. (c) Reliability — individual row failures don't prevent the rest of the page loading. Affects every Saved page load with 4+ companies.

**Effort:** M (2–6h)
**Confidence:** 4/5

## Item 9 — Pricing page FAQ uses `role="list"` / `role="listitem"` for an accordion, which is semantically incorrect

**Problem statement:** `PricingPage.tsx` lines 114–131 render the FAQ as a `<div role="list">` containing `<div role="listitem">` wrappers for each question. The interactive element inside each listitem is a `<button aria-expanded>` — making this an accordion, not a list. Screen readers that expose list role will announce "list, 4 items" and then announce each item without conveying the accordion semantics. The correct ARIA pattern for a disclosure accordion is no wrapping list role — just the `<button aria-expanded>` + `<div role="region" aria-labelledby>` pair, which is already correctly implemented for the inner elements.

**Evidence:**
- `frontend/src/pages/PricingPage.tsx` line 114: `<div style={{...}} role="list">`
- Line 116: `<div key={idx} role="listitem" ...>`
- Lines 118–127: Inner `<button aria-expanded aria-controls>` + `<div role="region" aria-labelledby>` — these are correct.
- The list/listitem wrapper adds incorrect semantics around a correctly-implemented accordion pattern.

**Proposed solution:** Remove `role="list"` from the outer `<div>` and `role="listitem"` from each item `<div>`. The inner button/region pair is already ARIA-correct for a disclosure accordion. No visual change required.

```tsx
// Before
<div style={{...}} role="list">
  {FAQS.map((item, idx) => (
    <div key={idx} role="listitem" style={{...}}>

// After
<div style={{...}}>
  {FAQS.map((item, idx) => (
    <div key={idx} style={{...}}>
```

**User impact:** Screen-reader users get correct accordion semantics (expand/collapse with state announced) instead of misleading list semantics. Small code change, meaningful accessibility improvement. Relevant for any user relying on AT to evaluate pricing before signing up.

**Effort:** S (< 2h)
**Confidence:** 5/5

## Item 10 — No "share to email" or "copy formatted summary" path in bulk check results

**Problem statement:** After a bulk check, the only output path is CSV export. Competitors (Red Flag Alert, Endole) provide a "send to colleague" or structured summary share that fits the actual workflow: a recruiter or finance lead runs the bulk check, then needs to share the "Avoid" and "Watch" rows with their team or record them in a ticket. A bulk CSV with 50 rows is less useful than a one-click formatted summary of the flagged items only.

**Evidence:**
- `frontend/src/pages/BulkCheckPage.tsx` lines 165–195: Only action available for results is CSV export of all rows.
- The existing `handleCopyLink` pattern in `CompanyReportPage.tsx` shows copy-to-clipboard is already a supported UX primitive in this codebase.
- The data to build the summary already exists: `result.rows.filter(r => r.bucket === 'avoid' || r.bucket === 'watch')` with name, number, risk level, score.

**Proposed solution:** Add a "Copy flagged summary" button next to the CSV export button in `BulkCheckPage.tsx`:

```tsx
<button
  className="btn btn-secondary btn-sm"
  onClick={() => {
    const flagged = result.rows.filter(r => r.bucket === 'avoid' || r.bucket === 'watch');
    const lines = flagged.map(r =>
      `${r.bucket?.toUpperCase()}: ${r.companyName ?? r.input} (#${r.companyNumber ?? '?'}) — Score ${r.score ?? '?'}/100`
    );
    const text = `CareerMove bulk check — ${new Date().toLocaleDateString('en-GB')}\n${lines.join('\n')}\nFull report: careermove.uk/app/bulk`;
    navigator.clipboard.writeText(text).then(() => setToast({ text: `Copied ${flagged.length} flagged companies`, tone: 'ok' }));
  }}
>
  <Icon name="copy" /> Copy flagged
</button>
```

This reuses the existing toast pattern, requires no new API calls, and works entirely from the in-memory `result` state.

**User impact:** Removes a workflow friction point for agency and freelancer users who need to communicate findings rather than file a CSV. Directly supports the Agency tier value proposition (team collaboration). The "Copy flagged" CTA also acts as a natural upsell moment for the waitlist.

**Effort:** S (< 2h)
**Confidence:** 4/5

## Competitor Context

**What Red Flag Alert does better:** Persistent "what changed" narrative (not just alert counts), AML/KYC compliance layer, sector-specific risk framing. CareerMove's opportunity: surface the `explanationSummary` (Item 1) and the disqualification detail already in the type system.

**What Creditsafe does better:** Director PEP/sanctions screening, real-time credit limit scores, global company data. CareerMove's response: transparency of methodology (already strong) + disqualification cross-check (already built for Agency tier). Not directly replicable without new data sources.

**What Endole does better:** 80+ enriched data fields including sector labels, inline SIC descriptions, and lead-gen filters. CareerMove's opportunity: SIC label lookup (Item 3) — a no-cost, no-API enrichment step.

**What all three do worse than CareerMove:** Opaque scoring, no plain-English framing of what the data means, no honest "we can't answer that" section. CareerMove's Methodology page and zone-tagged evidence are a genuine moat worth preserving.

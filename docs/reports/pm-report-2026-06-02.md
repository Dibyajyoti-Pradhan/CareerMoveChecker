## Item 1 — `addressSnippet` typed in `CompanySearchHit` but never rendered in search results

**Problem:** `CompanySearchHit.addressSnippet: string` is declared in `frontend/src/types/index.ts` (line 10) but is never referenced anywhere in the codebase. Every search result row in `SearchPage.tsx` shows company name, number, status, incorporation date, and type — but not the address. Users comparing two companies with similar names (e.g. "Deliveroo Operations Limited" vs "Deliveroo Plc") have no address signal in the list to disambiguate before clicking. Competitors including Companies House's own search UI and Endole surface the registered address in search results.

**Evidence:**
- `frontend/src/types/index.ts` line 10: `addressSnippet: string;` declared on `CompanySearchHit`
- `grep -rn "addressSnippet" frontend/src/` returns only the type declaration — zero usage anywhere
- `frontend/src/pages/SearchPage.tsx` lines 240–247: `ResultRow` renders `companyName`, `companyNumber`, `companyStatus`, `incorporatedOn`, `companyType` — `addressSnippet` never accessed
- Endole's company search (competitor) shows the registered address in every result row

**Proposed solution:** Add `addressSnippet` as a subtle third line in each search result's `.body` section, after the meta badges:

```tsx
// In ResultRow, after the .meta div:
{hit.addressSnippet && (
  <div style={{ fontSize: 12, color: 'var(--muted)', marginTop: 2 }}>
    {hit.addressSnippet}
  </div>
)}
```

No new API call needed — the field already arrives in the `searchCompanies` response.

**User impact:** All users on the search results page. Disambiguation between identically-named entities is a real daily problem — especially for agency users checking "Jones Consulting" where there may be dozens of results. Showing the address removes a wasted click-through. Reduces navigation errors and improves confidence in result selection.

**Effort:** S (under 2 hours — one JSX addition)
**Confidence:** 5/5

## Item 2 — `Charge.deliveredOn` typed but never rendered; delivery date silently dropped

**Problem:** The `Charge` interface in `frontend/src/types/index.ts` (line 50) includes `deliveredOn?: string` — the date the charge was delivered to Companies House for registration. This is a legally significant date: a charge only has priority over other creditors from the date of delivery, not creation. The Finance tab in `CompanyReportPage.tsx` shows `c.createdOn` but never accesses `c.deliveredOn`. For any user assessing secured lending risk, this omission gives an incomplete picture.

**Evidence:**
- `frontend/src/types/index.ts` line 50: `deliveredOn?: string;` on `Charge` interface
- `grep -rn "deliveredOn" frontend/src/` returns only the type declaration — zero usage
- `frontend/src/pages/CompanyReportPage.tsx` line 609: only `formatDate(c.createdOn)` is rendered; `c.deliveredOn` is never accessed
- Companies House's own public search UI shows both "Created" and "Delivered" dates on every charge record

**Proposed solution:** Extend the date display in the charge `.acc-row` to show both dates when `deliveredOn` is present:

```tsx
<div className="small muted">
  Created {formatDate(c.createdOn)}
  {c.deliveredOn && ` · Delivered ${formatDate(c.deliveredOn)}`}
</div>
```

**User impact:** Finance teams and agency users checking charges for due-diligence purposes. The delivery date determines priority ranking of charges — showing only the creation date is potentially misleading for anyone assessing secured creditor structure. Also signals to power users that CareerMove renders every available public field rather than cherry-picking.

**Effort:** S (under 2 hours — one conditional addition)
**Confidence:** 5/5

## Item 3 — Bulk "Refresh" button in SavedPage bulkbar has no `onClick` handler — dead interaction

**Problem:** In `SavedPage.tsx`, when one or more saved companies are selected, a "bulkbar" action toolbar appears. It contains a "Refresh" button (line 246) that renders `<Icon name="refresh" /> Refresh` but has **no `onClick` prop and no handler** of any kind. Clicking it does nothing. The individual per-row refresh (line 331) works correctly via `api.refreshReport(s.companyNumber)`, but the bulk equivalent is a dead element that creates false affordance. Users selecting multiple companies to refresh will click it repeatedly, see nothing happen, and lose trust.

**Evidence:**
- `frontend/src/pages/SavedPage.tsx` line 246: `<button className="btn btn-ghost btn-sm"><Icon name="refresh" /> Refresh</button>` — no `onClick`, no `disabled`, no handler anywhere in the file
- `frontend/src/pages/SavedPage.tsx` line 331: The individual per-row refresh is correctly wired: `await api.refreshReport(s.companyNumber).catch(() => null)`
- The `selected: Set<string>` state is already available in scope (line 41), making the fix straightforward

**Proposed solution:** Wire the bulk Refresh button to iterate over `selected` and call `api.refreshReport` for each, updating `reports` state on completion:

```tsx
<button
  className="btn btn-ghost btn-sm"
  onClick={async () => {
    const nums = [...selected];
    const results = await Promise.allSettled(
      nums.map((n) => api.refreshReport(n).catch(() => null))
    );
    const updates: Record<string, CompanyReport | null> = {};
    results.forEach((r, i) => {
      updates[nums[i]] = r.status === 'fulfilled' ? r.value : null;
    });
    setReports((cur) => ({ ...cur, ...updates }));
    setToast({ text: `Refreshed ${nums.length} companies`, tone: 'ok' });
  }}
>
  <Icon name="refresh" /> Refresh
</button>
```

**User impact:** Any user who selects multiple companies to refresh at once — a common workflow before a batch of contract renewals or placements. The dead button is a functional bug that also undermines trust in the bulkbar's other buttons (Compare, Export CSV).

**Effort:** S (under 2 hours — wiring an existing pattern)
**Confidence:** 5/5

## Item 4 — Hardcoded fake "1 hour ago" timestamp on every search result row

**Problem:** In `SearchPage.tsx` line 250, every search result row renders a "cached" timestamp using `relativeTime(new Date(Date.now() - 60 * 60 * 1000).toISOString())`. This is a hardcoded fake value — always exactly "1h ago" — with no connection to when the data was actually fetched. `CompanySearchHit` does not carry a `dataFetchedAt` field, so the timestamp cannot be real at the search-result level. The product's core brand promise is transparency and honesty ("We tell you which questions we can answer — and which we can't"). Showing a fabricated timestamp directly contradicts this.

**Evidence:**
- `frontend/src/pages/SearchPage.tsx` line 250: `<span className="cached">{relativeTime(new Date(Date.now() - 60 * 60 * 1000).toISOString())}</span>` — always computes to "1h ago" at render time
- `frontend/src/types/index.ts` lines 5–12: `CompanySearchHit` has no `dataFetchedAt` or `updatedAt` field
- The individual company report (`CompanyReportPage.tsx`) correctly shows `relativeTime(report.dataFetchedAt)` using a real timestamp from the API — that is where freshness should be surfaced

**Proposed solution:** Remove the `<span className="cached">` entirely from `ResultRow`. Optionally replace with a neutral static label:

```tsx
// Delete line 250 entirely. Optionally replace with:
<span className="cached" style={{ color: 'var(--muted-2)', fontSize: 11 }}>Live</span>
```

**User impact:** Every user who sees search results. Showing "1h ago" for a live API call creates false confidence about data age and contradicts the site's honesty-by-design positioning. Removing it (or replacing with "Live") is more honest and removes a potential trust cliff when users notice the timestamp never changes between searches.

**Effort:** S (under 30 minutes — one line deletion or replacement)
**Confidence:** 5/5

## Item 5 — Thumbs-down feedback sends `rating: 0` — ambiguous; conflicts with "no feedback given"

**Problem:** In `CompanyReportPage.tsx`, the thumbs-down button (`👎`) transitions to a comment state and then calls `api.submitFeedback({ rating: 0, ... })` (line 689). The thumbs-up button sends `rating: 1` (line 721). This means the API receives `rating: 0` for negative feedback — the same numeric value that represents "not set" or null in most data schemas. A backend analyst querying the feedback table will see `rating: 1` for positive and `rating: 0` for negative, which is indistinguishable from "no rating submitted at all" and makes thumbs-down versus skip unresolvable in analytics.

**Evidence:**
- `frontend/src/pages/CompanyReportPage.tsx` line 689: `rating: 0` submitted for negative feedback
- `frontend/src/pages/CompanyReportPage.tsx` line 721: `rating: 1` for positive feedback
- `frontend/src/api/client.ts` lines 61–67: `submitFeedback` accepts `rating: number` with no documented scale
- Analytics queries on the feedback table will conflate "skipped" (no call made) with "thumbs down" (`rating: 0`) since both look like absence of signal

**Proposed solution:** Change the thumbs-down submission to send `rating: -1`:

```tsx
// Line 689 — change:
rating: 0,
// to:
rating: -1,
```

One constant change. The API client accepts `number`, so no type change is needed.

**User impact:** Internal product and analytics team. Correctly polarised feedback data enables accurate measurement of "what % of reports users found useful" — a core product health metric. The current data undercounts negative feedback by conflating it with "no opinion given."

**Effort:** S (under 30 minutes — one constant change)
**Confidence:** 5/5

## Item 6 — Compare page autocomplete dropdown has no Escape-to-close or blur-to-dismiss

**Problem:** In `ComparePage.tsx`, the company-search autocomplete dropdown (lines 117–136) appears when suggestions are returned. There is no mechanism to close it without selecting an item: no `onBlur` handler, no Escape key handler, no click-outside detection. Once suggestions appear, pressing Escape does nothing — the dropdown stays open. This is a standard UX failure for any typeahead component. It also represents a WCAG 2.1 SC 1.4.13 (Content on Hover or Focus) violation: popup content triggered by keyboard focus must be dismissible without moving focus.

**Evidence:**
- `frontend/src/pages/ComparePage.tsx` lines 117–136: Suggestions `<div>` renders when `suggestions.length > 0` with no close mechanism
- `grep "onBlur\|Escape\|setSuggestions" frontend/src/pages/ComparePage.tsx` — only three `setSuggestions([])` calls, none triggered by blur or keydown; all triggered by item selection or short input
- The input element (lines 109–113) has no `onBlur` or `onKeyDown` handler

**Proposed solution:** Add two handlers to the input element:

```tsx
<input
  value={input}
  onChange={(e) => setInput(e.target.value)}
  placeholder="Add by company name or 8-digit number"
  onKeyDown={(e) => {
    if (e.key === 'Escape') { setSuggestions([]); }
  }}
  onBlur={() => {
    // Small delay to allow click on suggestion to fire first
    setTimeout(() => setSuggestions([]), 150);
  }}
/>
```

**User impact:** Any user on the compare page who types a company name and then decides not to add a suggestion. The dropdown currently traps focus context and cannot be dismissed by keyboard. This is also a WCAG 1.4.13 compliance gap relevant to a UK-facing B2B product.

**Effort:** S (under 2 hours — two event handler additions)
**Confidence:** 5/5

## Item 7 — `INFO` severity risk flags styled identically to `WARNING` — both render as orange `deduced` badge

**Problem:** In `CompanyReportPage.tsx` line 399, the severity badge on each risk flag uses: `f.severity === 'POSITIVE' ? 'direct' : 'deduced'`. This means `INFO`, `WARNING`, and `CRITICAL` all receive the `.zone-tag.deduced` CSS class — which renders in orange/amber (`var(--warn)` colour). `INFO` flags are informational notices (e.g. "under 2 years old — limited history"), not warnings. Showing them in the same orange as genuine `WARNING` and `CRITICAL` flags creates alarm where none is warranted. Competitors Endole and Creditsafe use neutral grey for informational notices and reserve amber/red for genuine risk signals.

**Evidence:**
- `frontend/src/pages/CompanyReportPage.tsx` line 399: `cn('zone-tag', f.severity === 'POSITIVE' ? 'direct' : 'deduced')` — INFO, WARNING, CRITICAL all map to `deduced` (orange)
- `frontend/src/index.css` lines 225–228: `.zone-tag.deduced { background: var(--warn-bg); color: var(--warn); }` — orange/amber
- `frontend/src/types/index.ts` line 2: `type RiskFlagSeverity = 'POSITIVE' | 'INFO' | 'WARNING' | 'CRITICAL';` — four distinct values; only POSITIVE maps uniquely
- Note: the icon logic on line 394 does correctly distinguish `CRITICAL || WARNING` → warn icon vs others → check icon — the icons are right, but the badge colours are not

**Proposed solution:** Expand the `zone-tag` class mapping to cover all four severities:

```tsx
<span className={cn(
  'zone-tag',
  f.severity === 'POSITIVE' && 'direct',
  f.severity === 'INFO' && 'not',
  (f.severity === 'WARNING' || f.severity === 'CRITICAL') && 'deduced',
)}>
  {f.severity}
</span>
```

This reuses the existing `.zone-tag.not` class (grey/muted) for INFO, keeping WARNING and CRITICAL as amber.

**User impact:** Every user reading the flags tab. `INFO` badges currently appear orange-alarming alongside `WARNING` and `CRITICAL` badges, making it impossible to scan-read which flags are serious versus contextual. A company with 2 `INFO` flags and 0 `WARNING` flags should not visually alarm the user in the same way as one with 2 `WARNING` flags. Fixing the colour directly improves decision quality.

**Effort:** S (under 2 hours — one conditional expression change)
**Confidence:** 5/5

## Item 9 — `useSeo` never updates Open Graph tags per-page; shared company links always show generic site metadata

**Problem:** The `useSeo` hook in `frontend/src/lib/seo.ts` only sets `document.title`, `meta[name="description"]`, and `link[rel="canonical"]`. It never touches Open Graph tags (`og:title`, `og:description`, `og:url`) or Twitter Card tags. These are set once, statically, in `frontend/index.html` to the generic site-level values (`"CareerMove"`, `"Should you trust this UK company? 60-second plain-English check."`). When a user copies the company report URL (a core CTA in the answer card — "Copy link") and pastes it into Slack, LinkedIn, or Teams, the link preview shows the generic homepage copy — not the company name or verdict. This breaks the social-sharing flow that the product actively encourages. Additionally, `og:image` is entirely absent from `index.html` — no social preview image exists at all.

**Evidence:**
- `frontend/src/lib/seo.ts` lines 20–31: Only `document.title`, `meta[name=description]`, and `link[rel=canonical]` are set — no OG/Twitter tag manipulation
- `frontend/index.html` lines 11–21: Static OG and Twitter tags; `og:title = "CareerMove"`, `og:description = "Should you trust this UK company?..."` — never overwritten by any page
- `frontend/src/pages/CompanyReportPage.tsx` lines 40–44: `useSeo` is called with a per-company title and description — but neither reaches OG tags
- `frontend/src/pages/CompanyReportPage.tsx` lines 169–175: "Copy link" CTA is prominently featured in the answer card — sharing is an explicit product feature
- `grep "og:image\|twitter:image" frontend/` — no results; social image meta tag is entirely absent

**Proposed solution:** Extend `useSeo` to also update OG and Twitter tags:

```typescript
// In seo.ts, inside the useEffect, after setting document.title:
const ogTitle = getOrCreate('meta[property="og:title"]', 'meta', { property: 'og:title' });
ogTitle.setAttribute('content', title);

if (description) {
  const ogDesc = getOrCreate('meta[property="og:description"]', 'meta', { property: 'og:description' });
  ogDesc.setAttribute('content', description);
  const twitterDesc = getOrCreate('meta[name="twitter:description"]', 'meta', { name: 'twitter:description' });
  twitterDesc.setAttribute('content', description);
}

if (canonical) {
  const ogUrl = getOrCreate('meta[property="og:url"]', 'meta', { property: 'og:url' });
  ogUrl.setAttribute('content', canonical);
}
```

Also add a static fallback `og:image` to `index.html` pointing to a placeholder (even an SVG favicon path) — any image is better than no social preview image.

**User impact:** Every user who shares a company report link via Slack, Teams, or LinkedIn. The "Copy link" button is prominently featured in the answer card — making the resulting share link show the company name and verdict instead of the generic site title is a direct improvement for word-of-mouth growth. A link to "ACME Corp — probably yes · CareerMove" previewing as "CareerMove · Should you trust this UK company?" looks like a broken or generic share.

**Effort:** M (half day — seo.ts extension + og:image static asset addition)
**Confidence:** 5/5

## Item 10 — Recent searches list has no per-item remove control and no "Clear all"

**Problem:** `SearchPage.tsx` maintains a `recent` list of up to 5 companies in `localStorage` (`RECENT_KEY`). The list is rendered with navigation buttons but has no mechanism to remove individual items or clear all. Once 5 slots are full, the oldest is silently rotated out — but a user who visited a wrong company (e.g. the wrong "Jones Consulting") has no way to remove that entry from the list. The list also has no "Clear all" header control. Competitors including the Companies House service and Endole both offer clearable history.

**Evidence:**
- `frontend/src/pages/SearchPage.tsx` lines 13–19: `getRecentSearches()` and `addRecentSearch()` exist but there is no `removeRecentSearch()` function
- `frontend/src/pages/SearchPage.tsx` lines 113–125: The recent search list renders each row as a `<button>` with only an `arrow-right` icon — no `x` or remove control
- No "Clear all" button exists in the recent search header
- `frontend/src/components/Icon.tsx` line 9: The `x` icon is available and already used throughout the app

**Proposed solution:** Add a `removeRecentSearch` helper function and two UI additions:

```tsx
function removeRecentSearch(number: string) {
  const cur = getRecentSearches().filter((r) => r.number !== number);
  localStorage.setItem(RECENT_KEY, JSON.stringify(cur));
}

// Update the recent-mini header to include a Clear all:
<div style={{ display: 'flex', justifyContent: 'space-between', alignItems: 'center', marginBottom: 12 }}>
  <h5 style={{ margin: 0 }}>Recent searches</h5>
  <button
    onClick={() => { localStorage.removeItem(RECENT_KEY); setRecent([]); }}
    style={{ background: 'transparent', border: 0, fontSize: 11, color: 'var(--muted)', cursor: 'pointer', fontFamily: 'var(--mono)' }}
  >
    Clear all
  </button>
</div>

// Add an x button to each rrow:
<button
  onClick={(e) => { e.stopPropagation(); removeRecentSearch(r.number); setRecent(getRecentSearches()); }}
  style={{ background: 'transparent', border: 0, color: 'var(--muted)', cursor: 'pointer', padding: 2, borderRadius: 4 }}
  aria-label={`Remove ${r.name} from recent searches`}
>
  <Icon name="x" size={12} />
</button>
```

**User impact:** All users who use the search page more than once. Privacy-conscious users want to clean up their local history. Power users who hit a wrong company want to remove it to keep the list meaningful. The inability to clear recent searches is a persistent friction point affecting every repeat user of the search flow.

**Effort:** M (half day — new helper function + two UI additions + state refresh)
**Confidence:** 5/5

## Executive Summary

This run surfaced 10 actionable opportunities across four themes: (1) typed data that is defined in the API but never rendered in the UI, (2) mobile breakpoint gaps that break real usability on phones, (3) UX dead-ends and copy quality issues that cost conversions, and (4) lightweight but high-value feature additions that competitors offer and that can be built purely from existing mock data. All items are achievable within the current in-memory mock — no new API endpoints required. Six are S-effort (under 2 hours each), four are M-effort.

## Item 1 — `explanationSummary` field is typed but never rendered

**Problem:** The `RiskAssessment` type includes an `explanationSummary: string` field (a longer narrative summary of the engine's reasoning) that is never displayed anywhere in the UI. Users see only the shorter `verdict` string and the `topReasons` list. The fuller narrative — which is what distinguishes CareerMove from a raw Companies House dump — sits invisible.

**Evidence:**
- `frontend/src/types/index.ts` line 108: `explanationSummary: string;` declared on `RiskAssessment`
- `frontend/src/pages/CompanyReportPage.tsx` lines 279–293: Zone B renders `a.verdict` (short string) and `a.topReasons.slice(0,4)` — `a.explanationSummary` is never accessed anywhere in this file or any other page file.
- Grep across all page files confirms zero usage of `explanationSummary`.

**Proposed solution:** In Zone B of `CompanyReportPage.tsx`, after the `answer-verdict` paragraph, add the `explanationSummary` as a secondary paragraph styled in `var(--ink-2)` at a slightly smaller size. Gate it with a "Read more" expand if the summary is long (> 300 chars), otherwise show inline. Example drop-in:

```tsx
{a.explanationSummary && (
  <p style={{ fontSize: 14, color: 'var(--ink-2)', lineHeight: 1.6, marginTop: 8 }}>
    {a.explanationSummary}
  </p>
)}
```

This is a one-liner addition. No new API calls needed — the field already arrives in the existing `getReport` response.

**User impact:** Every user who opens a report gets a richer plain-English narrative. Directly supports the product's core promise of "no jargon." Reduces bounce from the answer card because the verdict feels less thin.

**Effort:** S (< 2 hrs)  
**Confidence:** 9/10

## Item 2 — Disqualification match detail (reason, dates) typed but never shown

**Problem:** `DisqualificationMatch` has three optional detail fields — `reason?: string`, `disqualifiedFrom?: string`, `disqualifiedUntil?: string` — that are defined in the type but never rendered. When a match is found, the disqualification banner in Zone B shows only the officer name. The user has no visibility into *why* they were disqualified, *when* the ban started, or *when* it expires. This is arguably the most serious data point on the report and it is shown incompletely.

**Evidence:**
- `frontend/src/types/index.ts` lines 113–116: `DisqualificationMatch` interface with `reason`, `disqualifiedFrom`, `disqualifiedUntil` all typed.
- `frontend/src/pages/CompanyReportPage.tsx` line 270: The MATCH branch only renders `m.name` — the three detail fields are never accessed anywhere.
- Competitor Red Flag Alert (per research) shows ban reason and duration prominently in their KYB check output.

**Proposed solution:** Expand the disqualification banner's MATCH case into a small detail block. When `matches` has entries, list each match as a sub-row with name, reason (if present), and the from/until dates. The `disq-banner.bad` class already gives the visual container; just add inner markup:

```tsx
{report.disqualificationCheck.status === 'MATCH' && (
  <div className="disq-banner bad">
    <Icon name="alert" />
    <div>
      <div style={{ fontWeight: 600, marginBottom: 6 }}>
        {report.disqualificationCheck.matches.length} disqualified officer(s) on the current board
      </div>
      {report.disqualificationCheck.matches.map((m, i) => (
        <div key={i} style={{ fontSize: 13, marginTop: 4 }}>
          <b>{m.name}</b>
          {m.reason && <span> · {m.reason}</span>}
          {m.disqualifiedFrom && (
            <span> · Banned from {formatDate(m.disqualifiedFrom)}
              {m.disqualifiedUntil ? ` to ${formatDate(m.disqualifiedUntil)}` : ' (ongoing)'}
            </span>
          )}
        </div>
      ))}
    </div>
  </div>
)}
```

**User impact:** Recruiters and freelancers making a hiring decision need to know not just *that* a director is banned but *why* and *for how long*. This data is already in the mock — exposing it turns a one-line warning into genuinely actionable intelligence. Significant for Agency-tier users.

**Effort:** S (< 2 hrs)  
**Confidence:** 9/10

**User impact:** Improves completeness of the Officers panel. Useful for anyone using the report to verify a specific individual (freelancers checking a client's directors, agencies doing KYC lite). Positions CareerMove as more complete than the raw CH website.

## Item 4 — Note-on-save is a typed API feature with zero UI entry point

**Problem:** `SavedCompany.note?: string` is typed in `frontend/src/types/index.ts` (line 158). The API client exposes `api.updateSavedNote(companyNumber, note)` at `frontend/src/api/client.ts` lines 53–57. The CSV export in `SavedPage.tsx` even includes the `Note` column (line 237–242). Yet there is zero UI for a user to *write* a note — no input, no inline edit, no modal trigger anywhere in `SavedPage.tsx`. The feature is 80% built but completely invisible.

**Evidence:**
- `frontend/src/types/index.ts` line 158: `note?: string` on `SavedCompany`
- `frontend/src/api/client.ts` lines 53–57: `updateSavedNote` method exists and is wired
- `frontend/src/pages/SavedPage.tsx` lines 237–242: CSV export column header "Note" with `s.note ?? ''`
- `frontend/src/pages/SavedPage.tsx`: No `<input>` or `<textarea>` for note entry anywhere in the file — grep confirms zero usage of `updateSavedNote`

**Proposed solution:** Add an inline note edit to each saved-company row. Clicking a small pencil icon (already in the `Icon` set as `'edit'`) opens an `<input>` in place that calls `api.updateSavedNote` on blur or Enter. This avoids needing a modal. The row's `.actions` div already has icon buttons — add one more:

```tsx
// In the srow actions div:
<button className="icon-btn" title="Add note" onClick={() => setEditingNote(s.companyNumber)}>
  <Icon name="edit" size={14} />
</button>
{editingNote === s.companyNumber && (
  <input
    autoFocus
    defaultValue={s.note ?? ''}
    onBlur={(e) => {
      api.updateSavedNote(s.companyNumber, e.target.value).catch(() => {});
      setEditingNote(null);
    }}
    style={{ fontSize: 12, border: '1px solid var(--line)', borderRadius: 6, padding: '3px 6px' }}
    placeholder="Add a note…"
  />
)}
```

**User impact:** This unlocks a natural workflow: "Saved for final interview — check again before offer." Agencies tracking multiple clients especially need per-company notes. The CSV export already anticipates this column, so it becomes immediately useful in reporting workflows. High perceived value for near-zero backend cost.

**Effort:** M (2–4 hrs, needs local `editingNote` state slot)  
**Confidence:** 9/10

## Item 5 — `id-card` action row wraps badly on mobile, making the report unusable

**Problem:** On screens narrower than ~600px, the `.id-actions .row` in Zone A of the company report contains six buttons (Refresh, Print, Compare, Copy link, Companies House) in a `display: flex; flex-wrap: wrap` row. There is no `@media` rule that collapses or hides these on mobile. The result is a cramped multi-line button cluster that overflows the card on narrow viewports. The report is the product's single most important page and it is broken on mobile.

**Evidence:**
- `frontend/src/pages.css` line 377: `.id-actions { display: flex; flex-direction: column; gap: 8px; align-items: flex-end; }`
- `frontend/src/pages.css` line 379: `.id-actions .row { gap: 6px; }` — no `flex-wrap: wrap` containment or mobile override
- `frontend/src/pages.css` lines 662–685: The `@media (max-width: 1000px)` block covers `.id-meta` and `.flag-grid` but never mentions `.id-card`, `.id-actions`, or the action row
- `frontend/src/pages/CompanyReportPage.tsx` lines 222–249: Six action buttons rendered in a single `<div className="row">`

**Proposed solution:** Add a mobile override in the existing `@media (max-width: 1000px)` block in `pages.css`. Also collapse the secondary actions (Print, Compare, Copy) into a "More…" overflow menu on small screens, keeping only Refresh and Companies House visible. Minimal CSS change first:

```css
@media (max-width: 1000px) {
  .id-card { grid-template-columns: 1fr; }
  .id-actions { align-items: flex-start; }
  .id-actions .row { flex-wrap: wrap; gap: 6px; }
}
```

This alone will prevent overflow. A further S-effort improvement is to add `display: none` to the lower-priority buttons below 600px.

**User impact:** Any user opening a report on a smartphone currently sees a broken layout. Given the "check before signing an offer" use case, mobile access during commute or in a meeting is a primary scenario. This is a blocking usability issue.

**Effort:** S (< 2 hrs — CSS-only fix)  
**Confidence:** 10/10

## Item 6 — Bulk check row checkboxes have no select-all and no action wired

**Problem:** In `BulkCheckPage.tsx`, every result row renders `<input type="checkbox" />` (line 188) — but the checkbox has no `checked` prop, no `onChange` handler, and no connection to any state. It is a decorative HTML element with no function. There is also no "select all" checkbox in the table header. Users clicking these checkboxes see nothing happen — a confusing dead interaction.

**Evidence:**
- `frontend/src/pages/BulkCheckPage.tsx` line 188: `<input type="checkbox" />` with no props
- `frontend/src/pages/BulkCheckPage.tsx` lines 10–17: State declared for `result`, `loading`, `filename`, `filter`, `dragOver`, `toast` — no `selected` set defined
- `frontend/src/pages/BulkCheckPage.tsx` line 183: Header row `<div className="head"><span>#</span><span></span>...` — the second `<span></span>` is where a select-all checkbox should be

**Proposed solution:** Wire the checkboxes to a `selected: Set<string>` state (identical pattern to `SavedPage.tsx` which already implements this correctly). Add a select-all checkbox in the header and wire the "Export CSV" button to export only selected rows when a selection exists. This is a copy of the working pattern from `SavedPage`:

```tsx
const [selected, setSelected] = useState<Set<string>>(new Set());
const toggleRow = (n: string) => setSelected(cur => {
  const next = new Set(cur);
  next.has(n) ? next.delete(n) : next.add(n);
  return next;
});
// In header:
<input type="checkbox"
  checked={filtered.length > 0 && filtered.every(r => r.companyNumber && selected.has(r.companyNumber))}
  onChange={() => { /* toggle all */ }}
/>
// In brow:
<input type="checkbox"
  checked={r.companyNumber ? selected.has(r.companyNumber) : false}
  onChange={() => r.companyNumber && toggleRow(r.companyNumber)}
/>
```

**User impact:** Agencies running bulk checks want to select a subset of rows (e.g. only "AVOID" companies) to export or compare. The checkbox column currently creates false affordance — users click and nothing happens, which destroys trust. This is both a functional bug and a conversion risk.

**Effort:** M (2–4 hrs — state wiring + select-all logic)  
**Confidence:** 10/10

## Item 7 — Filing history capped at 20 with no "load more" — silent truncation

**Problem:** `report.filings.slice(0, 20)` in `CompanyReportPage.tsx` silently caps the displayed filing history at 20 entries. There is no visible count of total filings, no indicator that the list is truncated, and no "Show all" or pagination control. For a company with an extensive filing history (e.g. a 10-year-old plc), the user has no idea they are seeing only a fraction of the record.

**Evidence:**
- `frontend/src/pages/CompanyReportPage.tsx` line 579: `{report.filings.slice(0, 20).map((f) => (` — hard-coded slice with no UI affordance
- No count banner, no "showing X of Y" label, no expand control in the surrounding markup (lines 575–588)

**Proposed solution:** Add a `showAllFilings` boolean state, defaulting to `false`. Show the first 20 by default but display a count banner and a "Show all N filings" button when `report.filings.length > 20`. When clicked, toggle `showAllFilings` to render all. No new API call needed.

```tsx
const [showAllFilings, setShowAllFilings] = useState(false);
const visibleFilings = showAllFilings ? report.filings : report.filings.slice(0, 20);

// In the filings panel:
<h3>Recent filings {report.filings.length > 20 && (
  <span style={{ fontSize: 12, color: 'var(--muted)', fontWeight: 400 }}>
    (showing {visibleFilings.length} of {report.filings.length})
  </span>
)}</h3>
<div className="timeline">
  {visibleFilings.map((f) => ( ... ))}
</div>
{report.filings.length > 20 && !showAllFilings && (
  <button className="btn btn-ghost btn-sm" style={{ marginTop: 12 }}
    onClick={() => setShowAllFilings(true)}>
    Show all {report.filings.length} filings
  </button>
)}
```

**User impact:** Compliance-focused users (finance teams, agency due-diligence) frequently need the full history — especially to check for strike-off filings or capital reductions that may appear in older entries. Silent truncation is the kind of thing that causes a bad outcome and then a support complaint. Simple fix with real protective value.

**Effort:** S (< 2 hrs)  
**Confidence:** 9/10

## Item 9 — ComparePage has no SIC code or company type row, missing key decisional data

**Problem:** The compare grid in `ComparePage.tsx` shows eight rows (Score, Verdict, Trading history, Insolvency, Filings on time, Outstanding charges, Officers, Disqualified officers) but omits two decision-critical data points that are available in every `CompanyReport`: **SIC code** (industry sector) and **company type** (Ltd vs. Plc vs. LLP). Comparing a sole-trader LLP to a listed Plc on the same score scale without flagging the type difference gives a false apples-to-apples impression.

**Evidence:**
- `frontend/src/pages/ComparePage.tsx` lines 192–244: The eight `<Row>` components — no row for `r.profile.companyType` or `r.profile.sicCodes`
- `frontend/src/types/index.ts` lines 73–74: `companyType: string` and `sicCodes?: string[]` on `CompanyProfile`
- `frontend/src/pages/CompanyReportPage.tsx` line 258: SIC codes are shown in Zone A of the individual report

**Proposed solution:** Add two new rows to the compare grid using the existing `<Row>` component pattern — they require no new data fetching, just reading already-loaded fields:

```tsx
<Row label="Company type" cells={reports.map((r) => ({
  tone: 'ok' as const,
  val: r.profile.companyType,
  sub: '',
}))} />

<Row label="Industry (SIC)" cells={reports.map((r) => ({
  tone: 'ok' as const,
  val: r.profile.sicCodes?.join(', ') || '—',
  sub: '',
}))} />
```

Position after the "Trading history" row, before "Insolvency."

**User impact:** Agencies comparing multiple clients need to know if they are comparing like-for-like. A recruitment agency placing in financial services must know they are comparing FCA-regulated entities. This is a free data enrichment from fields already in the mock, and it makes the compare feature competitive with Creditsafe's company profile comparison tool.

**Effort:** S (< 2 hrs — two `<Row>` additions)  
**Confidence:** 9/10

## Item 10 — `AppNav` mobile collapses tab labels but leaves icon-only nav with no `title` text fallback visible

**Problem:** On mobile (< 1000px), the CSS rule `frontend/src/index.css` line 318 hides all `.app-tabs span:not(.dot-unread)` — meaning the icon labels ("Search", "Compare", "Saved", "Bulk check") are hidden, leaving only SVG icons. The `<NavLink>` elements for Search and Compare have a `title` attribute, but the `<NavLink>` for Saved uses `aria-label` conditionally (only when `unread > 0`) and the `<NavLink>` for Bulk check has a `title` attribute. However none of the icons have a visible `<span>` tooltip or native tooltip on touch devices, so mobile users with no context have four mystery icons in the navigation. The `aria-label` only covers screen-reader users — sighted mobile users see unlabelled icons.

**Evidence:**
- `frontend/src/index.css` line 318: `.app-tabs span:not(.dot-unread) { display: none; }` inside `@media (max-width: 1000px)`
- `frontend/src/layout/AppNav.tsx` lines 33–59: Four `<NavLink>` elements — only the Saved link has a conditional `aria-label`, the others only have `title` props
- The Save link at line 41: `aria-label` is only set when `unread > 0` — when there are no unread alerts, the Saved tab has neither visible text nor `aria-label` on mobile
- Icon SVGs all have `aria-hidden="true"` (Icon component line 70), correctly hiding them from assistive tech, but also meaning screen readers see nothing for unlabelled links

**Proposed solution:** Two changes: (1) Add unconditional `aria-label` to all four nav links for screen-reader coverage. (2) On mobile, render visually-hidden but screen-reader-readable `<span className="sr-only">` labels, or switch from `display: none` to `font-size: 0` trick that keeps layout labels. Simplest fix is to add persistent `aria-label` to every tab:

```tsx
// Search tab:
<NavLink to="/app/search" aria-label="Search UK companies" title="Search UK companies" ...>
// Compare tab:
<NavLink to="/app/compare" aria-label="Compare companies" title="Compare companies" ...>
// Saved tab (unconditional aria-label):
<NavLink to="/app/saved" 
  aria-label={unread > 0 ? `Saved — ${unread} unread alerts` : 'Saved companies'} ...>
// Bulk:
<NavLink to="/app/bulk" aria-label="Bulk check" title="Bulk check" ...>
```

Also add a `.sr-only` CSS class and apply it to the label spans so they remain in the accessibility tree without being visible:

```css
.sr-only { position: absolute; width: 1px; height: 1px; padding: 0; margin: -1px; overflow: hidden; clip: rect(0,0,0,0); white-space: nowrap; border: 0; }
```

**User impact:** WCAG 2.1 Success Criterion 2.4.6 (Headings and Labels) and 1.1.1 (Non-text Content) compliance. The Saved tab with zero unread alerts is currently a link with no accessible name — a clear WCAG AA failure. For sighted mobile users the mystery icons create friction (especially Compare and Bulk which are less common icons). Fixing this is also a legal risk mitigation for a UK-based B2B SaaS.

**Effort:** S (< 2 hrs — HTML attribute additions + one CSS class)  
**Confidence:** 10/10
